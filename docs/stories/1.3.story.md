## Story

**As a** presenter,
**I want** the system to extract the generated HTML code from Stitch API responses,
**so that** it can be deployed as a standalone website.

## Acceptance Criteria

1. Parse Stitch response to extract `htmlCode.downloadUrl` from generated screens
2. Fetch the HTML content from the download URL
3. Ensure HTML is self-contained (inline CSS, CDN fonts only)
4. If HTML references external assets, inline them or use CDN equivalents
5. Store generated HTML temporarily for deployment
6. `GET /api/result/:id` returns the generated HTML for preview

## Dependencies

- Story 1.1 (Express Server with Stitch MCP Proxy) — needs the Stitch API proxy and response handling

## Tasks / Subtasks

- [ ] Parse Stitch API response structure (AC: 1)
  - [ ] Identify `htmlCode.downloadUrl` field in response
  - [ ] Handle cases where field is missing or malformed
- [ ] Fetch HTML content from download URL (AC: 2)
  - [ ] HTTP GET request to download URL
  - [ ] Handle timeouts and retries
- [ ] Validate and sanitize HTML (AC: 3, 4)
  - [ ] Check that CSS is inline
  - [ ] Verify only CDN font references (Google Fonts)
  - [ ] Identify and inline any external asset references
  - [ ] Replace external assets with CDN equivalents where possible
- [ ] Implement temporary HTML storage (AC: 5)
  - [ ] In-memory or file-based storage with unique ID
  - [ ] Clean up old entries to prevent memory leaks
- [ ] Create `GET /api/result/:id` endpoint (AC: 6)
  - [ ] Return stored HTML by ID
  - [ ] Return 404 for unknown IDs
  - [ ] Set proper Content-Type headers

## Dev Notes

**Epic:** Epic 1 — Core Pipeline: Sketch to Deployed Website

**Technical Context:**
- Stitch API returns screen data including `htmlCode.downloadUrl` for generated screens
- Generated HTML must be self-contained: inline CSS, no external dependencies beyond CDN fonts
- NFR6: Generated websites MUST be valid HTML5, responsive, and include all CSS inline (single-file deployment)
- Storage is temporary — only needs to persist for the duration of a demo session
- This story bridges the gap between Stitch generation (Story 1.1) and Vercel deployment (Story 1.4)

**Stitch Response Structure:**
- The Stitch MCP `generate_screen_from_text` returns screen objects
- Each screen contains `htmlCode` with a `downloadUrl` property
- The download URL points to the complete HTML file

### Testing

- Manual testing with actual Stitch API responses
- Verify HTML is valid and self-contained
- Test `GET /api/result/:id` endpoint
- No unit tests required for MVP
